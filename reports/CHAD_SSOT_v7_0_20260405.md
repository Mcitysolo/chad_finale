## 0. Executive Summary

CHAD is an automated investment system that watches financial markets around the clock and decides when to buy and sell. Think of it as a tireless analyst that monitors stocks, futures, currencies, cryptocurrencies, and options simultaneously, following strict rules about when to trade and how much risk to take. It never acts on emotion, never sleeps, and never exceeds its risk budget. The system currently manages roughly $998,000 in a paper (simulated) trading account, proving itself before real money is committed.

Technically, CHAD is a multi-strategy algorithmic trading platform running on an AWS m5.large instance in ca-central-1. It comprises 14 independent strategy brains — each a deterministic, I/O-free signal generator — feeding into a unified execution pipeline that routes orders to Interactive Brokers (equities, futures, options) and Kraken (crypto). The system enforces a 50/30/20 capital chassis (alpha strategies/beta savings/adaptive hedges), implements 10 independent safety gates before any live order can execute, and maintains hash-chained NDJSON audit trails for every simulated fill. The intelligence layer uses a tiered AI stack (Ollama phi3:mini for fast local inference, Claude Haiku/Sonnet for synthesis, GPT-4 as fallback) to generate daily reports, advisory responses, and confidence adjustments — but AI can never place trades or modify configuration.

As of April 5, 2026, CHAD is in DRY_RUN mode with 14 registered strategies, 584 passing tests, and all P0-P2 hardening plus GAP-1 through GAP-25 governance items complete. The system has been paper-trading since early March 2026 with a win rate of 55.2% across 1,515 effective trades. The Shadow Confidence Router (SCR) is in CONFIDENT state with sizing factor 1.0 — no trading restrictions active.

The governing architectural principle is three-plane separation: the Intelligence Plane (AI advisory) is strictly separated from the Trading Plane (signal generation and execution) and the Safety Plane (gates, locks, and circuit breakers). AI advises; code trades; safety governs. No plane can override another.

## 2. Architectural Principles

### 2.1 Three-Plane Model

**Intelligence Plane** — The AI advisory layer. Contains Claude, Ollama, and GPT clients; advisory engines; daily report generators; Telegram bot responses. This plane CAN: synthesize market data into natural language, compute confidence adjustments, generate research summaries, answer operator questions. This plane CANNOT: place orders, modify configuration files, change risk parameters, access broker APIs, or override safety gates.

**Trading Plane** — The algorithmic signal and execution layer. Contains 14 strategy brains, the signal router, execution adapters (IBKR, Kraken), and the paper evidence writer. This plane CAN: generate TradeSignal intents from market data, route signals through policy gates, submit orders to brokers, record fills. This plane CANNOT: modify its own risk parameters, bypass safety gates, change strategy weights, or override the Safety Plane.

**Safety Plane** — The governance and risk layer. Contains LiveGate (10 gates), Profit Lock (5 tiers), SCR (confidence routing), the 50/30/20 chassis, config snapshots, and the Pending Actions pipeline. This plane CAN: block any trade, reduce position sizes, halt all trading, enforce capital allocation limits, detect configuration tampering. This plane CANNOT: generate trading signals or place orders — it only governs.

### 2.2 The Cardinal Rule

**Claude AI is Intelligence Plane only.** Specifically:

Claude CAN:
- Generate daily reports and morning briefs
- Answer operator questions via Telegram
- Compute confidence bias adjustments (range: [-0.15, +0.10])
- Synthesize macro context and research
- Suggest trades in advisory mode (operator must review)
- Write and modify code (via Claude Code, with operator approval)

Claude CANNOT:
- Place, modify, or cancel any order
- Edit `runtime/operator_intent.json` or any safety gate file
- Modify `config/strategy_weights.json` directly
- Restart services or change systemd configurations
- Access broker credentials or API sessions
- Override Profit Lock, SCR, or LiveGate decisions

### 2.3 Fail-Closed Philosophy

Every component defaults to the safest possible state when something goes wrong:

| Component | Fail-Closed Behavior |
|-----------|---------------------|
| LiveGate | If any gate file is missing, stale, or unreadable → DENY_ALL |
| Profit Lock | If equity is unknown → INACTIVE (no lock, but no sizing increase) |
| SCR | If win rate data insufficient → WARMUP (paper only) |
| Orchestrator | If allocation overlay fails → fall back to base weights |
| Chassis | If unknown strategy in weights → zero its allocation |
| Strategy brains | If context missing (bars, prices) → return empty signal list |
| IBKR Adapter | If contract qualification fails → log warning, do not submit |
| Config Snapshot | If any config file unreadable → fail the snapshot (do not skip) |
| Redis state bus | If Redis down → continue with JSON file polling (slower but safe) |
| AI tier | If Claude unavailable → fall back to Ollama/GPT; if all fail → skip advisory |

### 2.4 Governance Pipeline

All configuration changes follow this flow:

1. **Prepare** — Create a Pending Action JSON file in `control/pending_actions/` with `action_id`, `kind`, `payload`, `expires_ts_utc`, and `idempotency_key`.
2. **Approve** — Operator reviews and moves the action file to `control/approved_actions/`.
3. **Apply** — The ActionApplier (`chad/ops/action_applier.py`) scans `approved_actions/`, validates the payload, and atomically writes the new config (e.g., `config/strategy_weights.json`).
4. **Archive** — Applied actions move to `control/applied_actions/`. Expired actions move to `control/expired_actions/`.
5. **Audit** — The `action_applier_state.json` tracks all applied action IDs with timestamps. Config snapshots record before/after hashes.

Supported action kinds: `weight_rebalance_config` (strategy weight changes), `rebalance_execute` (portfolio rebalance trigger).

No direct file edits to config or runtime files. Ever. The only path to changing CHAD's behavior is through this pipeline.

## 4. Capital Chassis — 50/30/20

### 4.1 Sleeve Definitions

| Sleeve | Target | Strategies |
|--------|--------|-----------|
| **ALPHA** | 50% | alpha, alpha_futures, alpha_options, gamma, gamma_futures, gamma_reversion |
| **BETA** | 30% | beta |
| **ADAPTIVE** | 20% | omega, omega_macro, omega_vol, delta, delta_pairs, crypto |

The ALPHA sleeve contains all momentum and mean-reversion strategies — the "offense" of the portfolio. The BETA sleeve is the structural savings layer — long-term legend-driven allocation that should never be raided by short-term strategies. The ADAPTIVE sleeve contains hedging, macro, volatility, and market-neutral strategies — the "defense and diversification" of the portfolio.

### 4.2 Enforcement Mechanism

The `enforce_chassis()` function in `chad/risk/dynamic_risk_allocator.py` (line 404) runs on every orchestrator cycle:

1. Normalize all weights to sum to 1.0.
2. Compute actual sleeve totals (alpha_total, beta_total, adaptive_total).
3. Check if any sleeve has drifted more than `CHASSIS_TOLERANCE` (5%) from its target.
4. If enforcement is needed: scale each sleeve to its target while preserving intra-sleeve proportions. A strategy that was 60% of the ALPHA sleeve before enforcement remains 60% after.
5. Any strategy not assigned to any sleeve is zeroed — this prevents unregistered strategies from receiving capital.

Example: if overlay adjustments push ALPHA to 58% and BETA to 22%, enforcement scales ALPHA back to 50% and BETA back to 30%, preserving relative weights within each sleeve.

### 4.3 Why This Matters

The BETA sleeve exists to protect savings. Without chassis enforcement, aggressive alpha strategies could gradually absorb capital away from the structural long-term allocation — especially during strong trending markets when alpha strategies show high returns and attract more capital. This is a classic institutional problem: the trading desk raids the investment book. The 30% BETA floor prevents this. Even if every alpha strategy is performing brilliantly, BETA retains its 30% structural allocation. Conversely, during drawdowns, the ADAPTIVE sleeve's 20% ensures hedging and defensive strategies always have capital available.

### 4.4 Override

Set `CHAD_CHASSIS_ENFORCEMENT=0` to disable enforcement entirely. This should only be done during:
- Backtesting (where you want to test pure weight allocation without sleeve constraints)
- Emergency rebalancing (where you need to temporarily concentrate capital)
- Development (when testing new strategies that aren't yet assigned to a sleeve)

Never disable in production without understanding the consequences. Re-enable immediately after the override purpose is complete.

## 5. Execution Architecture

### 5.1 Signal Flow

The complete path from strategy idea to broker fill:

```
Strategy Brain (e.g., alpha.py)
    ↓ emits TradeSignal(strategy, symbol, side, size, confidence, asset_class, meta)
Signal Router / Aggregator
    ↓ merges signals from multiple strategies, deduplicates
Policy Gates (symbol cap gate, position guard, signal guard)
    ↓ filters signals that violate position limits or duplicate cooldowns
LiveGate Evaluation (10 gates)
    ↓ determines mode: ALLOW_LIVE / EXIT_ONLY / DENY_ALL
Execution Router
    ↓ routes by asset_class: equity→IBKR, futures→IBKR, options→IBKR, crypto→Kraken
IBKR Adapter (_intent_from_routed_signal)
    ↓ normalizes to NormalizedIntent(sec_type, exchange, currency, quantity, ...)
Contract Resolver (_ContractResolver.resolve)
    ↓ qualifies contracts with broker (Stock, Future, Option, ComboLeg, Forex)
Order Factory
    ↓ builds MKT or LMT order with what-if flag in dry_run mode
Broker Submission (ib.placeOrder or dry_run log)
    ↓ 10-second threading timeout on all broker calls
Fill Confirmation
    ↓ SubmittedOrder returned with ib_order_id, status, fill_price
Paper Evidence Writer
    ↓ writes hash-chained NDJSON records to fills, fees, execution_metrics ledgers
```

### 5.2 Execution Lanes

| Lane | sec_type | Resolver | Contract Type | Use Case |
|------|----------|----------|--------------|----------|
| **STK** | `STK` | `_resolve_stock()` | `Stock()` | Equities and ETFs (SPY, QQQ, AAPL, etc.) |
| **FUT** | `FUT` | `_resolve_future()` | `Future()` | Micro futures (MES, MNQ, MCL, MGC, ZN, ZB) |
| **OPT** | `OPT` | `_resolve_option()` | `Option()` | Individual options contracts (calls/puts) |
| **BAG** | `BAG` | `_resolve_combo()` | `Contract(secType=BAG)` + `ComboLeg[]` | Vertical spreads (Phase 6b — atomic 2-leg combo) |
| **CASH** | `CASH` | `_resolve_forex()` | `Forex()` | FX pairs (EURUSD, GBPUSD, etc.) |

The BAG lane (added in Phase 6b) eliminates naked leg risk for options spreads by submitting both legs as a single IBKR combo order. The broker guarantees fill-or-kill on both legs simultaneously.

All lanes enforce whole-unit quantities for sec_types in `default_whole_unit_sec_types = ("STK", "FUT", "OPT", "BAG")`. CASH (forex) uses fractional step sizing.

### 5.3 LiveGate — 10 Gates

All 10 gates are evaluated on every execution attempt. Any single gate failure results in DENY_ALL (fail-closed).

| # | Gate | State File | TTL | Blocks When |
|---|------|-----------|-----|-------------|
| 0 | Execution Config | env vars | — | Mode is not "live" |
| 1 | STOP Signal | `runtime/stop_state.json` | 24h | `stop == true` (operator kill-switch) |
| 1.5 | Profit Lock | `runtime/profit_lock_state.json` | 120s | `stop_new_entries == true` (HARD_STOP tier) |
| 2 | Operator Intent | `runtime/operator_intent.json` | 2h | Not exactly `"ALLOW_LIVE"` |
| 3 | Live Readiness | `runtime/live_readiness.json` | 600s | `ready_for_live != true` |
| 4 | Execution Quality | `runtime/execution_quality.json` | 600s | `env_label == "dangerous"` or `"unknown"` |
| 5 | SCR State | `runtime/scr_state.json` | 600s | `paper_only == true` or `state == "PAUSED"` |
| 6 | Mutation State | `runtime/action_state.json` | 120s | Pending actions > 0, rejected > 0, or lockout active |
| 7 | Change Canary | `runtime/change_canary_state.json` | 120s | `canary_factor < 1.0` or canary period active |
| 8 | Reconciliation | `runtime/reconciliation_state.json` | 300s | `status != "GREEN"` (broker/ledger mismatch) |
| 9 | Lifecycle Truth | `runtime/positions_truth.json` + `trade_lifecycle_state.json` | 120s | `truth_ok != true`, `gap_flag == true`, or `backlog_flag == true` |

**Final decision:** `ALLOW_LIVE` requires all gates passing AND `exec_mode == "live"` AND `ibkr_enabled` AND `!ibkr_dry_run`. Otherwise `EXIT_ONLY` (if exits allowed) or `DENY_ALL`.

### 5.4 P0 Hardening (Completed)

Four critical execution fixes applied during hardening phase:

**P0-1: reqContractDetails removed from hot paths.** IBKR's `reqContractDetails` call has unpredictable latency (sometimes >2s). It was called on every order submission for contract validation. Moved to one-time resolution at startup. Hot path now uses cached contract objects.

**P0-2: 10-second threading timeouts.** All broker API calls (`qualifyContracts`, `placeOrder`, `reqPositions`, etc.) are wrapped in `_call_with_timeout(fn, timeout_s=10)` using daemon threads. If any call exceeds 10 seconds, a `BrokerTimeoutError` is raised and logged. This prevents hung IBKR sessions from blocking the orchestrator loop.

**P0-3: Single IB() session discipline.** The adapter maintains exactly one `IB()` connection, reused across all submissions. Previously, multiple sessions could be created during reconnection, causing port conflicts and order ID collisions. Now enforced via `ensure_connected(force=True)` with explicit cleanup of stale sessions.

**P0-4: Paper execution attribution fixed.** The `StrategyAttributionEngine` uses a 5-tier resolution to ensure every fill is attributed to a real strategy (alpha, beta, gamma, etc.), never to placeholder values like "unknown" or "paper_exec". If no real strategy can be resolved, it raises `StrategyAttributionError` rather than silently misattributing.

### 5.5 Paper vs Live Execution

What changes when going from `DRY_RUN` to `LIVE`:

| Aspect | Paper (DRY_RUN) | Live |
|--------|-----------------|------|
| Order submission | Logged but not sent to broker | Sent to broker via `ib.placeOrder()` |
| Fill prices | Simulated from market data | Actual broker fills |
| LiveGate | Always DENY_ALL (gate 0 blocks) | All 10 gates must pass |
| Profit Lock | Computes but cannot block | Actively blocks at HARD_STOP tier |
| NDJSON ledger | Fills written to paper evidence files | Fills written to live evidence files |
| Idempotency | Active (prevents duplicate dry_run logs) | Active (prevents duplicate live orders) |
| What-if simulation | Available via `simulate_what_if_in_dry_run` | Not used |
| Contract qualification | Optional (configurable) | Required (must qualify before submission) |

What stays the same: signal generation, strategy logic, dynamic caps computation, chassis enforcement, config snapshot hashing, SCR evaluation, and all safety gates (they just also allow LIVE when all conditions are met).

## 6. Intelligence Layer

### 6.1 AI Tier Stack

| Tier | Model | Use Case | Cost | Latency |
|------|-------|----------|------|---------|
| 1 (local) | Ollama phi3:mini | Fast local inference, macro filtering, quick classification | Free | ~200ms |
| 2 (fast cloud) | Claude Haiku | Strategy intelligence, advisory responses, daily report sections | ~$0.001/query | ~1-2s |
| 3 (deep cloud) | Claude Sonnet | Complex synthesis, research reports, detailed analysis | ~$0.01/query | ~3-5s |
| 4 (fallback) | GPT-4 | Fallback when Claude unavailable | ~$0.03/query | ~5-10s |

The tier stack is fail-through: if tier 1 is unavailable, fall to tier 2; if Claude is unavailable, fall to GPT-4; if all AI is unavailable, skip advisory entirely (fail-closed — no advisory is safer than wrong advisory).

### 6.2 Strategy Intelligence

**Confidence bias:** The intelligence layer can adjust strategy confidence by [-0.15, +0.10]. A macro headwind might reduce alpha confidence by 0.15; a strong sector trend might boost gamma confidence by 0.10. This bias is advisory — it adjusts the confidence field of TradeSignals but cannot create or suppress signals entirely.

**Universe filtering:** The intelligence layer can suggest removing symbols from a strategy's universe based on earnings risk, liquidity concerns, or sector rotation. This is implemented via `control/constraints/` files that the strategy engine reads.

**Regime profiles:** Each strategy has documented best/worst regimes. The intelligence layer monitors current market conditions and can flag when a strategy is operating in its worst regime, prompting the operator to consider reducing its allocation.

**Fail-closed contract:** If Claude/GPT/Ollama are all unavailable, all confidence biases default to 0.0 (no adjustment), no universe filtering occurs, and no advisory content is generated. Strategies continue operating on their base logic without AI input.

### 6.3 Advisory Engine

Six specialized engines in `chad/intel/`:

| Engine | File | Purpose |
|--------|------|---------|
| Research | `research_engine.py` | Synthesize market data into research summaries |
| Lessons | `lessons_engine.py` | Extract patterns from past trades for learning |
| Risk Explainer | `risk_explainer.py` | Plain-English risk exposure breakdown |
| Macro Interpreter | `macro_interpreter.py` | Translate macro indicators into actionable context |
| Model Doctor | `model_doctor.py` | Diagnose strategy performance issues |
| Synthetic Analyst | `synthetic_analyst.py` | Generate analyst-style commentary on positions |

The `advisory_engine.py` (50KB) orchestrates these engines and routes operator questions to the appropriate specialist. Synthesis combines outputs into coherent responses delivered via Telegram.

### 6.4 Daily Reports

**Daily Report** — Generated at 4:35 PM ET by `chad/ops/daily_chad_report.py`:
1. **Header** — Date and system identification
2. **Did We Make Money?** — P&L summary in plain English
3. **What Did We Do?** — Trade activity summary with instrument translations (MES → "S&P 500 futures")
4. **Best/Worst Move** — Highlight trades with explanation
5. **Strategy Status** — Per-strategy performance with plain names (alpha → "momentum", beta → "long-term holds")
6. **Market Temperature** — Volatility and regime summary
7. **CHAD Status** — System health (gates, SCR, profit lock)
8. **Paper Account** — Equity, positions, and allocation
9. **Sign Off** — Next actions and watchlist

**Morning Brief** — Generated at 8:55 AM ET by the same module with `morning` argument. Contains: overnight market moves, pre-market conditions, today's watchlist, active positions update, and any alerts.

Both reports are sent to Telegram and designed for a non-technical reader — the operator should understand everything without knowing what EMA or ATR mean.

### 6.5 Safety Constraints

The intelligence layer is bound by these hard constraints:

1. Cannot place, modify, or cancel any order.
2. Cannot edit any file in `runtime/` (safety gate state files).
3. Cannot edit any file in `config/` (must go through Pending Actions).
4. Cannot access broker API credentials or sessions.
5. Cannot restart or stop any systemd service.
6. Cannot modify strategy source code at runtime.
7. Confidence bias is hard-clamped to [-0.15, +0.10] — cannot zero out or double a strategy.
8. Universe filtering suggestions are advisory only — operator must approve.
9. Telegram bot is read-only — cannot write to `operator_intent.json`.
10. All AI outputs are logged but never automatically acted upon.

### 8.2 Redis State Bus

`chad/core/state_bus.py` implements a Redis Pub/Sub state bus for sub-second state propagation:

| Channel | Purpose |
|---------|---------|
| `chad:stop_signal` | Emergency stop broadcast (<100ms propagation) |
| `chad:dynamic_caps` | Capacity updates (strategy dollar caps) |
| `chad:live_gate` | Trading readiness gate state changes |
| `chad:profit_lock` | Profit protection tier changes |
| `chad:fast_loop` | High-frequency loop heartbeat |
| `chad:live_readiness` | System readiness status |

**STOP propagation:** When `StateBusPublisher.publish_stop(reason)` is called, all subscribers receive the stop signal within ~100ms via Redis Pub/Sub. The `live_loop.py` registers an `_on_stop()` handler that immediately halts all trading.

**Fallback contract:** If Redis is unavailable, `publish()` returns `False` silently. All state files continue to be written as JSON to `runtime/`. Consumers fall back to polling these files at their normal cadence (8s fast loop, 60s signal loop). This ensures CHAD never depends on Redis for correctness — only for speed.

### 8.3 Tiered Loop

Two concurrent async loops run via `asyncio.gather()`:

**Fast Loop (8s)** — `_run_fast_loop()`:
- Checks portfolio snapshot freshness (broker sync age)
- Validates price cache availability (symbol count)
- Monitors bar cache completeness
- Reads profit lock state
- Writes `runtime/fast_loop_state.json` (cycle count, health flags)
- Publishes heartbeat to Redis `chad:fast_loop`
- Logs every 50 cycles (~7 minutes) to avoid log spam

**Signal Loop (60s)** — `run_forever()`:
- Calls `refresh_dynamic_caps()` to recompute per-strategy dollar budgets
- Builds portfolio snapshot from broker data
- Runs strategy allocation through chassis enforcement
- Writes `runtime/dynamic_caps.json` atomically
- Appends to `runtime/decision_trace_YYYYMMDD.ndjson` (audit trail)
- Publishes to Redis `chad:dynamic_caps`

Environment control: `CHAD_FAST_LOOP_ENABLED=1`, `CHAD_FAST_LOOP_INTERVAL_SECONDS=8`, `CHAD_ORCH_INTERVAL_SECONDS=60`.

### 8.4 Backtesting Engine

`chad/analytics/backtest_engine.py` provides zero-lookahead backtesting:

- `BacktestContext` mirrors the live `MarketContext` interface so strategies run identically in backtest and production.
- Bar iteration is strictly chronological — the engine feeds bars one at a time, and strategies can only see data up to the current bar.
- Fill simulation uses next-bar open (not current close) to prevent lookahead bias.
- Current limitation: daily bars only. Full intraday validation requires minute bars, which are available for a subset of instruments but not yet integrated into the backtest harness.

## 9. Governance and Safety

### 9.1 Pending Actions Pipeline

**Flow:**
1. Create action JSON in `control/pending_actions/` with required fields: `action_id`, `kind`, `status`, `payload`, `expires_ts_utc`, `idempotency_key`.
2. Operator reviews, then copies to `control/approved_actions/`.
3. `ActionApplier.main()` scans `approved_actions/`, validates, and applies.
4. Applied actions archived to `control/applied_actions/`. Expired actions to `control/expired_actions/`. Unsupported kinds to `control/unsupported_actions/`.
5. State tracked in `runtime/action_applier_state.json` (list of applied IDs + metadata).

**Supported action kinds:**
- `weight_rebalance_config` — writes validated weights to `config/strategy_weights.json`
- `rebalance_execute` — triggers portfolio rebalance with specified profile

**Idempotency:** Each action has a unique `idempotency_key`. The applier checks `applied_actions` list before applying — duplicate keys are silently skipped.

### 9.2 Config Snapshots

`chad/ops/config_snapshot.py` computes SHA-256 hashes of all files in `config/`:

- Each `config/*.json` file gets an individual hash.
- A composite hash is computed from all individual hashes (sorted by filename).
- Snapshots are stored in `data/config_snapshots/snapshot_YYYYMMDDTHHMMSSZ.json`.
- Used for tamper detection: if any config file changes between snapshots without a corresponding applied action, the change is flagged.
- The Change Canary gate (LiveGate gate 7) can use snapshot drift to block live trading.

### 9.3 Profit Lock

Five escalating tiers that progressively restrict trading as realized P&L grows:

| Tier | Mode | Trigger (% of equity) | Sizing Factor | New Entries |
|------|------|----------------------|---------------|-------------|
| 0 | NORMAL | < 1.5% | 1.0 | Allowed |
| 1 | WARN | 1.5% - 3.0% | 1.0 | Allowed (log warning) |
| 2 | LOCK1 | 3.0% - 5.0% | 0.50 | Allowed (half size) |
| 3 | LOCK2 | 5.0% - 8.0% | 0.25 | Allowed (quarter size) |
| 4 | LOCK3 | 8.0% - 10.0% | 0.10 | Allowed (tenth size) |
| 5 | HARD_STOP | >= 10.0% | 0.00 | **Blocked** |

The sizing factor multiplies `portfolio_risk_cap` in the dynamic allocator. At HARD_STOP, the cap drops to zero and LiveGate gate 1.5 blocks all new entries. Exit orders are always allowed (you must be able to close positions).

Thresholds are configurable via environment variables (`CHAD_PROFIT_LOCK_WARN_PCT`, etc.). The profit lock state is written to `runtime/profit_lock_state.json` with 60-second TTL.

### 9.4 SCR — Shadow Confidence Router

The SCR evaluates CHAD's overall trading performance to determine if the system should be trusted with progressively more responsibility:

| State | Meaning | Trading Allowed |
|-------|---------|-----------------|
| WARMUP | Insufficient data (< 2000 trades) | Paper only |
| PAUSED | Performance below threshold | Paper only |
| PAPER_ONLY | Performance marginal | Paper only |
| CONFIDENT | Performance meets thresholds | Paper + Live |

**Win rate calculation:** Computed over a rolling 2000-trade window. Only "trusted" trades count — trades flagged as `pnl_untrusted` (manual interventions, system errors, etc.) are excluded from the denominator.

**Current state:** CONFIDENT with win_rate=55.2%, sizing_factor=1.0, Sharpe-like=8.6, max_drawdown=-1.19%. Based on 1,515 effective trades (485 excluded as untrusted out of 2,000 paper trades).

**Death spiral fix** (tag: `PNL_DEATHSPIRAL_FIXED_20260404`): A bug was discovered where untrusted PnL records were dragging down the win rate calculation, pushing SCR into PAUSED, which prevented new trades, which prevented recovery. The fix was to exclude `pnl_untrusted` records from the win rate denominator entirely.

## 11. Known Limitations and Deferred Items

### 11.1 Active Limitations

1. **Paper-only track record.** All 1,515 trades are simulated. Paper fills use market data, not actual broker fills, which means slippage and market impact are not measured. Live trading will reveal the real execution quality gap.

2. **Daily bar backtesting only.** The backtest engine operates on daily bars. Intraday strategies (ALPHA, GAMMA) cannot be fully validated without minute-bar backtesting. Minute bars exist for some instruments but are not yet integrated.

3. **Single-server architecture.** CHAD runs on one m5.large instance. There is no failover, no hot standby, and no geographic redundancy. If the instance goes down, trading stops until it is restarted.

4. **No real-time P&L for paper.** Paper P&L is computed from fill records, not from broker-reported positions. Reconciliation against broker truth is implemented but requires live fills to be fully exercised.

5. **ALPHA_FOREX dormant.** The FX strategy is registered but not actively generating signals due to missing live FX data from Canada.

6. **Options chain data dependency.** ALPHA_OPTIONS relies on cached options chain data (`runtime/options_chains_cache.json`). If the chain provider service is down, no options signals are generated (fail-closed, but a gap).

### 11.2 Deferred Features

| Feature | Why Deferred | When to Revisit |
|---------|-------------|----------------|
| Voice coach | Nice-to-have; Telegram text is sufficient for now | After live trading is stable for 30 days |
| Databento market data | Premium alternative to IBKR native data; cost-prohibitive at current scale | When AUM exceeds $5M or IBKR data quality becomes insufficient |
| M6E subscription | Micro EUR futures need dedicated subscription in some regions | When OMEGA_MACRO is generating consistent signals on ZN/ZB first |
| Statistical arbitrage | More complex than DELTA_PAIRS; requires cointegration testing | After DELTA_PAIRS proves profitable in live trading |
| Multi-server HA | Single-server is fine for current AUM; HA adds operational complexity | When AUM exceeds $10M or uptime SLA becomes critical |
| Intraday backtest | Requires minute-bar integration into backtest harness | Next development cycle after live activation |

### 11.3 What Live Trading Will Reveal

The paper trading period cannot predict these real-world factors:

1. **Slippage and market impact.** Paper fills assume execution at theoretical prices. Live orders will experience bid-ask spread, partial fills, and price movement between signal and fill.

2. **IBKR execution latency under load.** The 10-second timeout protects against hangs, but real latency distribution under live order flow is unknown.

3. **Broker rejection patterns.** IBKR may reject orders for reasons not encountered in paper (insufficient margin, contract restrictions, regulatory limits).

4. **Correlation breakdown speed.** DELTA_PAIRS assumes stable correlations. Live markets may reveal that correlations break faster than the 60-day lookback can detect.

5. **Options fill quality.** BAG combo orders depend on both legs filling simultaneously. In illiquid options markets, the broker may reject the combo.

6. **Overnight gap risk.** Several strategies hold overnight positions. Gap risk is modeled in backtest but not stress-tested against black swan events.

**What to watch in the first 30 days of live trading:**
- Execution quality: compare planned vs actual fill prices
- Reconciliation state: GREEN should be maintained at all times
- SCR stability: win rate should not drop below 50% sustained
- Profit lock triggers: if LOCK1 fires frequently, reduce risk_budget_pct
- Strategy attribution: every fill must have a real strategy name (no "unknown")

## 13. Glossary

**ATR (Average True Range):** A measure of how much a stock's price typically moves in a day. Used to set stop-losses and filter for volatility regimes. Higher ATR = more volatile.

**BAG Order:** An IBKR combo order that submits multiple legs (e.g., buy one option and sell another) as a single atomic order. Either both legs fill or neither does. Eliminates "naked leg risk."

**Bollinger Band:** A price envelope drawn at a set number of standard deviations above and below a moving average. When price touches the band, it may be overextended.

**Chassis:** The 50/30/20 capital allocation framework that ensures no single strategy type can dominate the portfolio. Like the frame of a car — it holds everything in place.

**ComboLeg:** A component of an IBKR BAG order. Each leg specifies a contract, action (buy/sell), ratio, and exchange.

**Confidence:** A 0-to-1 score that a strategy assigns to each signal, indicating how certain the strategy is. Higher confidence = larger position size.

**Config Snapshot:** A SHA-256 hash of all configuration files, taken periodically to detect unauthorized changes.

**DTE (Days to Expiration):** How many calendar days remain before an options contract expires.

**DecisionTrace:** An NDJSON audit log recording every decision the orchestrator makes — what caps were computed, what weights were used, what the portfolio looked like. Used for post-mortem analysis.

**DRY_RUN:** The execution mode where CHAD simulates all trading without actually placing orders at the broker. Everything works identically to live trading except orders are logged rather than submitted.

**EMA (Exponential Moving Average):** A smoothed average of recent prices that gives more weight to recent data. Used by strategies to detect trends. Fast EMA crossing above slow EMA = bullish trend.

**EWMA (Exponentially Weighted Moving Average):** Similar to EMA but used for non-price data like portfolio weights. Smooths out noise in allocation decisions.

**Fail-Closed:** A design principle where if anything goes wrong, the system stops trading rather than continuing with potentially wrong data. The opposite of fail-open.

**LiveGate:** The 10-gate safety system that must unanimously approve before any live order can execute. One failed gate = no trading.

**NDJSON (Newline-Delimited JSON):** A file format where each line is a complete JSON object. Used for append-only ledgers (fills, fees, metrics). Each record is hash-chained to the previous one for tamper detection.

**Pending Action:** A proposed configuration change that must be explicitly approved before it takes effect. Part of the governance pipeline that prevents accidental or unauthorized changes.

**Profit Lock:** A 5-tier system that progressively restricts trading as daily realized profit grows. Prevents giving back gains.

**RSI (Relative Strength Index):** A momentum indicator ranging from 0-100. Above 70 is considered "overbought" (may fall); below 30 is "oversold" (may rise).

**Rollback:** Reverting the codebase to a known-good state using a git tag. Example: `git checkout RATIFICATION_MASTER_20260402`.

**SCR (Shadow Confidence Router):** A system that monitors CHAD's trading performance and decides whether to trust the system with real money. Like a probation officer for the algorithm.

**Sleeve:** A group of strategies that share a capital allocation target. ALPHA sleeve = growth strategies (50%), BETA sleeve = savings (30%), ADAPTIVE sleeve = hedging and alternatives (20%).

**Z-score:** How many standard deviations a value is from its mean. Z-score of +2.0 means the value is unusually high. Used by DELTA_PAIRS to detect when a price ratio has diverged from normal and is likely to revert.
