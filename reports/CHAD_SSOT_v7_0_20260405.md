## 0. Executive Summary

CHAD is an automated investment system that watches the markets around the clock, decides when to buy and sell, and manages risk — all without human intervention for routine decisions. Think of it as a team of specialized analysts that never sleep: one watches trends, another watches for crashes, another trades bonds, another trades crypto, and so on. There are 14 of these "analysts" (called strategies), each with its own job and its own slice of the portfolio. Right now, CHAD runs on practice money to prove it works before real dollars are committed.

Technically, CHAD is a multi-strategy algorithmic trading system built in Python 3.12, running on an AWS EC2 m5.large instance in ca-central-1. It connects to Interactive Brokers for equities, futures, and options execution; Kraken WebSocket for real-time crypto pricing; and an Anthropic Claude AI intelligence layer for macro analysis, confidence biasing, and operator advisory. The system is architected around three strict planes — Intelligence, Trading, and Safety — with 143 systemd services orchestrating market data ingestion, signal generation, execution planning, risk management, and observability. All runtime state is file-based with Redis pub/sub acceleration, and every component is fail-closed by design.

As of April 5, 2026, CHAD is paper trading with ~$998K in simulated capital across 14 live strategies (13 funded, 1 disabled). The system has passed all 4 mandatory chaos drills, 574 automated tests, and a 10-domain ratification proof pack (5 VERIFIED, 5 PARTIAL). The Shadow Confidence Router reports a 55.18% win rate against a 55% target, with a Sharpe-like metric of 8.6 and max drawdown of -1.18%. The system is approved for live activation pending operator GO.

The governing principle is absolute: Claude AI operates exclusively in the intelligence plane. It can advise, analyze, and recommend — but it can never execute trades, modify risk parameters, or bypass safety gates. The operator makes all final decisions. Every configuration change must pass through the Pending Actions governance pipeline. No exceptions.

## 2. Architectural Principles

### 2.1 Three-Plane Model

CHAD's architecture is divided into three strictly separated planes. No component may cross plane boundaries without going through defined interfaces.

**Intelligence Plane** — What CHAD thinks

The intelligence plane encompasses all AI-powered analysis: Claude API calls, advisory synthesis, strategy intelligence (confidence bias, universe filtering, regime classification), Google Trends signals, macro interpretation, and Telegram advisory responses. Components in this plane can read any runtime state file but can write only to advisory-specific output files (`runtime/strategy_intelligence.json`, `runtime/trends_state.json`, `reports/advisory/`). They cannot modify config files, operator intent, stop state, or any trading parameters. They produce recommendations — never actions.

**Trading Plane** — What CHAD does

The trading plane encompasses signal generation (14 strategies), execution planning (signal router, execution pipeline), broker communication (IBKR adapter, Kraken), position management, and trade ledger recording. Components here read strategy configs and risk caps, generate TradeSignal objects, route them through LiveGate, and submit orders to the broker. The trading plane operates on deterministic rules — never on AI output directly. AI confidence adjustments are bounded ([-0.15, +0.10]) and clamped before application.

**Safety Plane** — What CHAD cannot do

The safety plane encompasses LiveGate (10 independent gates), profit lock (5 escalating tiers), SCR (shadow confidence routing), stop state, operator intent, config snapshots with tamper detection, change canary, and reconciliation. Every component in this plane is fail-closed: if data is missing, stale, or corrupt, the answer is always DENY. The safety plane has absolute veto over the trading plane. No trade can execute without passing all 10 gates.

### 2.2 The Cardinal Rule

**Claude AI is intelligence plane only. It advises. It never executes.**

What Claude CAN do:
- Analyze market data and provide advisory text
- Adjust confidence bias within bounded range [-0.15, +0.10]
- Recommend universe filtering (avoid/prefer symbols)
- Classify market regime (risk_on/risk_off/neutral/crisis)
- Generate daily reports and morning briefs
- Answer operator questions via Telegram
- Produce research, macro, and risk explanations

What Claude CANNOT do:
- Execute trades or submit orders
- Modify config files or strategy weights
- Change risk parameters or position limits
- Override LiveGate or any safety gate
- Approve or apply pending actions
- Modify operator intent or stop state
- Access API keys or credentials
- Write to any file outside advisory/intelligence outputs

### 2.3 Fail-Closed Philosophy

Every subsystem defaults to the safest possible state when information is missing, stale, or corrupt.

| Subsystem | Fail-Closed Behavior |
|-----------|---------------------|
| **LiveGate** | Missing or stale state file → DENY_ALL |
| **Stop state** | Missing or expired TTL → `stop=true` |
| **Operator intent** | Missing → DENY_ALL |
| **SCR** | Missing → PAUSED (paper_only=true) |
| **Profit lock** | HARD_STOP tier → stop_new_entries=true |
| **Execution quality** | "dangerous" or "unknown" → DENY_ALL |
| **Reconciliation** | RED or missing → DENY_ALL |
| **Mutation state** | Pending actions exist → DENY_ALL |
| **Change canary** | Config drift detected → canary_factor<1.0 → DENY_ALL |
| **Strategy intelligence** | Claude unavailable → neutral bias (0.0) |
| **Signal router** | Exception during routing → empty signal list |
| **IBKR adapter** | No IB session injected → ConnectionError |
| **Broker timeout** | >10 seconds → BrokerTimeoutError |

### 2.4 Governance Pipeline

All configuration changes follow the Pending Actions pipeline. No direct file edits to governed config.

**Flow:**
1. Change is written as a JSON action file to `control/pending_actions/`
2. Operator reviews the action and moves it to `control/approved_actions/`
3. `chad-action-applier.service` scans approved queue and dispatches
4. Pre-apply: config snapshot is created (SHA256 hash of all config files)
5. Action is applied atomically (tmp file → fsync → rename)
6. Post-apply: action is moved to `control/applied_actions/`
7. Mutation state publisher updates `runtime/action_state.json`
8. Change canary validates composite hash against pre-apply snapshot

**Supported action kinds:**
- `weight_rebalance_config` — atomically updates `config/strategy_weights.json`
- `rebalance_execute` — runs rebalance executor in subprocess

**Idempotency guarantee:** Each action has a unique ID tracked in `runtime/action_applier_state.json`. Re-processing the same action ID is a no-op.

**LiveGate integration:** While `pending_count > 0` or `rejected_count > 0`, LiveGate returns DENY_ALL. No live trading during governance transitions.

### 4.2 Enforcement Mechanism

**File**: `chad/risk/dynamic_risk_allocator.py` (lines 404-477)

`enforce_chassis()` runs on every allocation cycle:

1. Maps each strategy to its sleeve (ALPHA/BETA/ADAPTIVE)
2. Sums actual allocation per sleeve
3. Compares to target (50/30/20)
4. If any sleeve drifts > 5% from target: proportional scaling applied
5. Intra-sleeve proportions are preserved (strategies within a sleeve maintain relative ratios)
6. Unknown/unmapped strategies are zeroed out
7. Final weights normalized to sum ~1.0

### 4.3 Why This Matters

The BETA sleeve is the safety net. It holds 25-30% of capital in long-term, low-churn positions that compound over time. Without chassis enforcement, aggressive alpha strategies could gradually eat into this allocation during winning streaks, leaving the portfolio overexposed when a correction hits. The 50/30/20 chassis ensures that no matter how well (or poorly) individual strategies perform, the portfolio always maintains structural diversification. It is the difference between a hedge fund and a gambling account.

### 4.4 Override

Set environment variable `CHAD_CHASSIS_ENFORCEMENT=0` to disable enforcement.

**When to use**: Only during controlled testing scenarios where you need to observe raw strategy weight behavior without chassis normalization. Never in production.

## 5. Execution Architecture

### 5.1 Signal Flow

Complete path from signal generation to broker fill:

```
Strategy handler (e.g., alpha.py)
    ↓ emits TradeSignal(strategy, symbol, side, size, confidence, asset_class, meta)
SignalRouter
    ↓ nets by symbol, deduplicates, produces RoutedSignal
ExecutionPipeline
    ↓ builds PlannedOrder objects, validates notional
LiveGate
    ↓ evaluates all 10 gates — ALLOW or DENY_ALL
IBKRAdapter
    ↓ normalizes to NormalizedIntent (strategy, sec_type, exchange, currency, side, qty)
ContractResolver
    ↓ resolves to ib_insync contract (Stock, Future, Option, Combo, Forex)
OrderFactory
    ↓ builds ib_insync Order (MKT/LMT, TIF, quantity)
Broker submission
    ↓ _call_with_timeout(ib.placeOrder, contract, order, timeout_s=10)
Fill confirmation
    ↓ SubmittedOrder with ib_order_id
NDJSON ledger
    → data/trades/trade_history_YYYYMMDD.ndjson (hash-chained, append-only)
```

**TradeSignal contract** (from `chad/types.py`):
```
strategy: StrategyName          — which strategy generated this
symbol: str                     — ticker symbol
side: SignalSide                — BUY or SELL
size: float                     — units (not dollars)
confidence: float               — 0.0-1.0 hint
asset_class: AssetClass         — EQUITY, ETF, FUTURES, OPTIONS, CRYPTO, FOREX, CASH
created_at: datetime            — when signal was created
meta: Dict[str, Any]            — diagnostic payload
```

### 5.2 Execution Lanes

**STK Lane (Stocks/ETFs)**
- Asset classes: equity, etf
- Contract: `Stock(symbol, exchange="SMART", currency="USD")`
- Primary exchange overrides for SPY, QQQ, IVV, VOO
- Whole units only

**FUT Lane (Futures)**
- Asset class: futures
- Contract: `Future(symbol, lastTradeDateOrContractMonth, exchange, currency, multiplier)`
- Requires `contract_month` in intent.meta (P0-1 hardening — no reqContractDetails in hot path)
- Whole units only
- Supported: MES, MNQ, MCL, MGC, ZN, ZB

**OPT Lane (Individual Options)**
- Asset class: options
- Contract: `Option(symbol, expiry, strike, right, exchange="SMART", currency="USD", multiplier="100")`
- Requires `expiry`, `strike`, `right` in meta
- Whole units only

**BAG Lane (Combo Spreads — Phase 6b)**
- Asset class: options (spread)
- Contract: `Contract(secType="BAG", comboLegs=[long_leg, short_leg])`
- Requires `expiry`, `long_strike`, `short_strike`, `long_right`, `short_right` in meta
- Atomic execution — both legs or neither (eliminates naked leg risk)
- Used by: alpha_options

**CASH Lane (Forex/Crypto)**
- Asset class: forex, cash
- Contract: `Forex(pair, exchange="IDEALPRO")`
- 6-letter symbol format (e.g., EURUSD) → base/quote split
- Fractional units (quantity_step=0.0001)

### 5.3 LiveGate — 10 Independent Gates

**File**: `chad/core/live_gate.py`

Every gate must pass for `allow_ibkr_live=true`. First failure = DENY_ALL.

| # | Gate | Source | TTL | Blocks When |
|---|------|--------|-----|-------------|
| 1 | **STOP** | `runtime/stop_state.json` | 24h | `stop=true` or missing/stale |
| 1.5 | **PROFIT_LOCK** | `runtime/profit_lock_state.json` | 120s | `stop_new_entries=true` |
| 2 | **OPERATOR_INTENT** | `runtime/operator_intent.json` | 2h | mode != ALLOW_LIVE |
| 3 | **LIVE_READINESS** | `runtime/live_readiness.json` | 600s | `ready_for_live=false` or missing |
| 4 | **EXECUTION_QUALITY** | `runtime/execution_quality.json` | 600s | env_label = "dangerous" or "unknown" |
| 5 | **SCR** | `runtime/scr_state.json` | 600s | `paper_only=true` or state=PAUSED |
| 6 | **MUTATION_STATE** | `runtime/action_state.json` | 120s | pending_count > 0 or rejected_count > 0 |
| 7 | **CHANGE_CANARY** | `runtime/change_canary_state.json` | 120s | canary_factor < 1.0 or canary active |
| 8 | **RECONCILIATION** | `runtime/reconciliation_state.json` | 300s | status != GREEN |
| 9 | **LIFECYCLE_TRUTH** | `runtime/positions_truth.json` + `trade_lifecycle_state.json` | 120s | truth_ok=false, gap_flag=true, or backlog_flag=true |
| 10 | **EXEC_MODE** | env or `runtime/live_mode.json` | — | exec_mode != "live" |

### 5.4 P0 Hardening (Completed April 2, 2026)

**P0-1: Remove reqContractDetails from hot paths** — Eliminated all implicit broker socket calls during contract resolution. Futures require `contract_month` in meta; options require `expiry/strike/right`. No contract qualification in the execution hot path.

**P0-2: 10-second execution liveness contract** — All broker calls wrapped in threading timeout. `_call_with_timeout(fn, timeout_s=10.0)` runs function in daemon thread with hard wall-clock deadline. Timeout raises `BrokerTimeoutError` with `failure_class=TIMEOUT`. Applied to: qualifyContracts, whatIfOrder, placeOrder, connect.

**P0-3: Single IB() session discipline** — No implicit `IB()` creation in hot path. Adapter requires pre-managed session via `ib_factory` dependency injection. Default lazy factory raises `ConnectionError` if not injected. Callers must create IB() once at startup, connect with retries, and pass to adapter.

**P0-4: Fix paper_exec attribution** — Never silently flatten strategy to "paper_exec". Raises `StrategyAttributionError` if strategy cannot be resolved. Attribution precedence: explicit strategy kwarg → source_strategies → planner artifact → tags → error (no fallback). Placeholder names ("", "unknown", "manual", "paper_exec") are never accepted as primary.

### 5.5 Paper vs Live Execution

| Aspect | Paper Mode | Live Mode |
|--------|-----------|-----------|
| **Account** | PAPER_EXEC | Real IB account |
| **Broker field** | "paper_exec" | "IBKR" |
| **is_live** | false | true |
| **Ledger** | data/fills/FILLS_YYYYMMDD.ndjson | data/trades/trade_history_YYYYMMDD.ndjson |
| **LiveGate** | Not required (gates skipped) | All 10 gates must pass |
| **Contract resolution** | Optional IB session | Required IB session |
| **Idempotency** | Enabled (SQLite) | Enabled (SQLite) |
| **Evidence** | Hash-chained fills + fees + metrics | Broker confirmation + NDJSON |

**Configuration switches**:
- `CHAD_EXECUTION_MODE`: "dry_run" (default), "paper", "live"
- `IBKR_ENABLED`: true/false (default true)
- `IBKR_DRY_RUN`: true (default) — when false, live orders are submitted

### 6.5 Safety Constraints

The intelligence layer is subject to these absolute constraints:

1. **No trade execution** — advisory outputs are text only, never order objects
2. **No config mutation** — cannot write to any file in `config/`
3. **No safety override** — cannot modify LiveGate, stop state, operator intent, profit lock
4. **No credential access** — API keys loaded by system, never exposed to AI prompts
5. **No unbounded influence** — confidence bias clamped to [-0.15, +0.10], applied through existing strategy logic only
6. **No persistence beyond advisory** — writes only to advisory-specific paths (strategy_intelligence.json, trends_state.json, reports/advisory/)
7. **Graceful degradation** — every AI call has a timeout, retry limit, and fallback; system operates normally without AI
8. **Token budget** — 100K tokens/day hard cap prevents runaway API costs
9. **Prompt isolation** — AI never sees other users' data, internal credentials, or system-level configs
10. **Read-only runtime access** — intelligence components read runtime state but never modify trading state files

### 8.3 Tiered Loop

**Signal loop (60s)**:
- Entry: `orchestrator.run_forever()`
- Runs `refresh_dynamic_caps()` — computes per-strategy dollar risk caps
- Writes `runtime/dynamic_caps.json`
- Appends to `data/traces/decision_trace_YYYYMMDD.ndjson`
- Publishes `chad:dynamic_caps` to Redis
- Configurable: `CHAD_ORCH_INTERVAL_SECONDS=60`

**Fast loop (8s)**:
- Checks: portfolio_snapshot freshness, price_cache freshness, ibkr_bars_cache freshness, profit_lock status
- Writes `runtime/fast_loop_state.json` (cycle_count, broker_ok, price_ok, etc.)
- Publishes `chad:fast_loop` to Redis
- Every 50 cycles (~7 min): logs heartbeat, writes `runtime/redis_state.json`
- Configurable: `CHAD_FAST_LOOP_INTERVAL_SECONDS=8` (clamped 1-30s)
- Enabled by: `CHAD_FAST_LOOP_ENABLED=true`

**Concurrency**: `await asyncio.gather(signal_loop, fast_loop)` — both run simultaneously. Fast loop errors are caught and logged; they never kill the signal loop.

**Live loop (60s)** — separate service (`chad/core/live_loop.py`):
1. Broker truth rebuild (reconcile position_guard against actual broker positions)
2. Strategy routing (evaluate all handlers, produce signals)
3. Intent planning (DecisionPipeline → StrategyEngine → PolicyEngine → SignalRouter → ExecutionPlan)
4. Per-intent guard evaluation (attribution, position guard, signal dedup with 10-min cooldown)
5. Execution (new position or flip)
6. Idle detection (CYCLE_IDLE if no signals)

**File**: `chad/analytics/backtest_engine.py`

4. No partial fill simulation

**Usage**: `python3 -m chad.analytics.backtest_engine --strategy alpha_futures --symbols MES MNQ MCL MGC --start 2025-06-01 --end 2026-04-01 --equity 100000`

## 9. Governance and Safety

### 9.1 Pending Actions Pipeline

**Directory structure**:
```
control/
  pending_actions/     — Actions awaiting operator review
  approved_actions/    — Human-approved, ready for execution
  applied_actions/     — Successfully executed (archive)
  expired_actions/     — Past TTL (archive)
  unsupported_actions/ — Unknown action kinds (archive)
```

**Flow**: Create JSON action → operator reviews → move to approved → ActionApplier dispatches → pre-apply config snapshot → atomic write → post-apply archive → mutation state published → change canary validated.

**Action kinds**: `weight_rebalance_config` (atomic strategy_weights.json update), `rebalance_execute` (subprocess rebalance).

**Idempotency**: `runtime/action_applier_state.json` tracks applied action IDs. Same ID is never re-executed.

**LiveGate integration**: `pending_count > 0` OR `rejected_count > 0` → DENY_ALL. No trading during governance transitions.

### 9.2 Config Snapshots

**File**: `chad/ops/config_snapshot.py`

- Snapshots all files in `config/*.json` (~16 files)
- Per-file SHA256 hash
- Composite hash: SHA256 of concatenated "filename:sha256" pairs
- Timestamped output: `data/config_snapshots/snapshot_YYYYMMDDTHHMMSSZ.json`
- **Pre-apply snapshot**: Created before every action execution (historical record)
- **Periodic check**: Mutation state publisher compares composite hash every 60s
- **Tamper detection**: Unexpected hash change → `change_canary_state.json` sets `canary_factor=0.5` → LiveGate DENY_ALL

### 9.3 Profit Lock

**File**: `chad/risk/profit_lock.py`

Five escalating tiers based on realized PnL as percentage of equity:

| Tier | PnL Threshold | Sizing Factor | Stops New Entries | Effect |
|------|--------------|---------------|-------------------|--------|
| NORMAL | < 1.5% | 1.0x | No | Full trading |
| WARN | >= 1.5% | 1.0x | No | Alert only |
| LOCK1 | >= 3.0% | 0.50x | No | Half position sizes |
| LOCK2 | >= 5.0% | 0.25x | No | Quarter position sizes |
| LOCK3 | >= 8.0% | 0.10x | No | 10% position sizes |
| HARD_STOP | >= 10.0% | 0.0x | **Yes** | No new entries, exits only |

**Equity providers** (fallback chain): dynamic_caps.json → positions_snapshot.json → portfolio_state.json → positions_truth.json.

**PnL aggregation**: Scans `data/trades/*YYYYMMDD*.ndjson`, extracts realized PnL, excludes paper trades, untrusted trades, and non-finite values.

**Integration**: Sizing factor multiplied into portfolio_risk_cap in Dynamic Risk Allocator. stop_new_entries gates LiveGate (Gate 1.5).

**Output**: `runtime/profit_lock_state.json` (decision + thresholds), `runtime/pnl_state.json` (PnL tracking).

### 9.4 SCR — Shadow Confidence Router

**File**: `chad/analytics/shadow_confidence_router.py`

Four states with escalating confidence:

| State | Condition | Sizing | Paper Only | Effect |
|-------|-----------|--------|------------|--------|
| WARMUP | < 50 effective trades | 10% | Yes | Insufficient data |
| PAUSED | Below cautious thresholds | 0% | Yes | Trading frozen |
| CAUTIOUS | Between cautious and confident | 25% | No | Reduced sizing |
| CONFIDENT | All thresholds met | 100% | No | Full trading |

**Threshold parameters**:
- Confident: win_rate >= 55%, Sharpe >= 0.70, max_drawdown >= -$2000
- Cautious: win_rate >= 45%, Sharpe >= 0.30, max_drawdown >= -$5000
- Warmup: effective_trades >= 50

**Current state** (April 2026): CONFIDENT — win rate 55.18%, Sharpe 8.6, max drawdown -1.18%, 1515 effective trades / 2000 total.

**pnl_untrusted exclusion**: Trades marked `pnl_untrusted=true` are excluded from effective trade count. This prevents unreliable P&L data from corrupting the confidence calculation.

**The death spiral fix** (tag: `PNL_DEATHSPIRAL_FIXED_20260404`): A bug caused untrusted trades (with PnL=0) to accumulate, diluting the win rate calculation, which dropped SCR to PAUSED, which stopped trading, which meant no new trades to improve the win rate — a self-reinforcing loop. Fixed by: (1) tagging untrusted trades explicitly, (2) excluding them from SCR calculations, (3) raising the sample cap from 100 to 2000 effective trades.

## 11. Known Limitations and Deferred Items

### 11.1 Active Limitations

1. **Paper-only execution** — All trading is simulated. Real broker behavior (slippage, partial fills, market impact) has not been validated.
2. **Daily bar backtesting** — Backtest engine uses daily bars only. Intraday dynamics, gap risk, and micro-structure effects are not modeled.
3. **No slippage model** — Paper fills use close price. Real execution will experience bid-ask spread, latency, and market impact.
4. **Bootstrap configs** — `config/risk.json`, `config/data_sources.json`, and `config/feature_flags.json` contain placeholder values pending dedicated audits.
5. **Options chain data** — Alpha_options depends on options chain availability from IBKR, which requires real-time options data subscription.
6. **M6E subscription** — omega_macro trades M6E (Micro Euro FX) which may need separate IBKR data subscription.
7. **SCR sample size** — 1515 effective trades from 2000 total. Statistical significance improves with more data. The 55% win rate is encouraging but not conclusive.

### 11.2 Deferred Features

| Feature | Reason Deferred | When to Revisit |
|---------|----------------|-----------------|
| **Voice coach** | Not essential for trading execution | After live activation and stable operation |
| **Databento** | Alternative market data provider — IBKR native sufficient for now | If IBKR data quality degrades |
| **M6E subscription** | $10/month CME bundle covers most; M6E may need separate | When omega_macro signals trigger frequently on M6E |
| **Stat-arb expansion** | delta_pairs covers initial pairs; broader stat-arb needs more correlated instruments | After 30 days of delta_pairs live performance |
| **Alpha_forex activation** | Full architecture exists but no proven edge in paper | When FX backtests show positive expectancy with slippage |
| **Intraday backtesting** | Requires 1-minute bar data collection at scale | After daily-bar strategies prove profitable live |
| **Autonomy bounds** | Config exists but execution disabled | After manual pilot phase proves allocation stability |

### 11.3 What Live Trading Will Reveal

1. **Slippage reality** — Paper fills at close price. Live fills will show real spread and impact costs, especially for illiquid instruments (MCL, MGC).
2. **Latency effects** — Paper has zero latency. Live execution through IB Gateway has measurable delay. The 10-second timeout (P0-2) may need tuning.
3. **Partial fills** — Paper fills are always complete. Live orders may partially fill, requiring position management logic.
4. **Margin requirements** — Paper uses simplified margin. Live margin calls could force unexpected liquidations.
5. **Futures roll behavior** — Paper doesn't model contract expiry and roll. Live futures positions need roll management.
6. **After-hours gaps** — Equity positions may gap significantly at open. Risk models based on close-to-close may underestimate overnight risk.
7. **Correlation breakdown** — Delta_pairs assumes stable correlations (0.94-0.99). Market stress may cause decorrelation.
8. **SCR calibration** — The 55% win rate target and sizing factors may need adjustment based on real P&L distribution.

**First 30 days checklist**: Monitor fill quality, actual slippage vs. paper assumption, strategy attribution accuracy, reconciliation frequency, and profit lock tier behavior with real P&L.

## 13. Glossary

| Term | Plain English Definition |
|------|------------------------|
| **ATR** | Average True Range — a measure of how much a price typically moves in a day. Higher ATR = more volatile. |
| **BAG order** | A combo order that bundles two options legs (buy and sell) into one atomic trade. Both execute together or neither does. Prevents "naked leg" risk. |
| **Bollinger Bands** | Price bands drawn above and below a moving average. When price touches the outer band, it may be overextended. |
| **Chassis** | The 50/30/20 allocation framework that protects portfolio structure. Like the frame of a car — it keeps everything in place. |
| **Config snapshot** | A timestamped SHA256 hash of all config files. Used to detect unauthorized changes (tamper detection). |
| **Confidence** | A 0-1 score representing how strongly a strategy believes in a trade. Higher = more conviction. Affects position size. |
| **DecisionTrace** | An append-only audit log recording every decision the orchestrator makes each cycle. Used for debugging and compliance. |
| **EMA** | Exponential Moving Average — a smoothed average of recent prices that gives more weight to the latest data. |
| **Fail-closed** | A design principle where missing or broken data results in the safest possible action (deny trading, reduce size, etc.). The opposite of fail-open. |
| **LiveGate** | The 10-gate safety system that must approve every live trade. All gates must pass. One failure = all trading blocked. |
| **NDJSON** | Newline-Delimited JSON — each line in the file is a complete JSON object. Used for trade ledgers and audit logs. |
| **Pending Action** | A proposed configuration change that sits in a queue until a human approves it. No config changes bypass this process. |
| **Profit Lock** | A five-tier system that automatically reduces position sizes (and eventually stops trading) as daily profits increase. Prevents giving back gains. |
| **Reconciliation** | The process of comparing what CHAD thinks it holds vs. what the broker actually reports. Must be GREEN for live trading. |
| **RSI** | Relative Strength Index — a 0-100 momentum indicator. Above 70 = overbought (might fall). Below 30 = oversold (might rise). |
| **SCR** | Shadow Confidence Router — a system that monitors CHAD's trading performance and adjusts position sizes based on win rate and risk metrics. Must be CONFIDENT for full trading. |
| **Sleeve** | A bucket within the 50/30/20 chassis. ALPHA (growth), BETA (preservation), ADAPTIVE (tactical/hedge). |
| **Stop state** | The emergency kill switch. When engaged, all trading halts immediately. |
| **TradeSignal** | The standard output from every strategy: symbol, direction (buy/sell), size, confidence, and metadata. |
| **Z-score** | A statistical measure of how far a value is from its average, measured in standard deviations. Z-score of 2 = very unusual. Used in pairs trading. |
| **Operator intent** | A file-based control that tells CHAD whether it's allowed to trade live. Modes: DENY_ALL, EXIT_ONLY, ALLOW_LIVE. |
| **Idempotency** | The guarantee that submitting the same order twice produces only one execution. Tracked via SQLite database. |
| **Regime** | The current market environment classification: risk_on (bullish), risk_off (bearish), neutral, or crisis. Different strategies perform differently in different regimes. |
| **VIX** | The CBOE Volatility Index — often called the "fear gauge." Below 15 = calm. Above 25 = worried. Above 30 = panic. |
| **Point value** | The dollar value of one point of price movement in a futures contract. MES = $5/point, MCL = $100/point. |
| **Half-life** | In pairs trading, the expected number of days for a diverged ratio to return halfway to its mean. Shorter half-life = faster reversion. |
| **Conviction score** | A multi-factor score (0-1) used by the delta strategy combining vol expansion, breakout, momentum, event risk, and liquidity signals. |
| **Hash chain** | A sequence where each record contains the hash of the previous record. Proves no records were inserted, deleted, or modified after the fact. Used in trade ledgers. |
| **Canary** | A change-detection mechanism. If config files change unexpectedly, the canary triggers and blocks live trading until the operator investigates. Named after canaries in coal mines. |
| **Backoff** | When an API call fails, waiting progressively longer before retrying (e.g., 2s, 4s, 8s). Prevents overwhelming a struggling service. |
| **TTL** | Time To Live — how long a piece of data is considered fresh. After TTL expires, the system treats the data as stale and may fail-closed. |
