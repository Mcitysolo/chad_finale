Think of it as a team of 14 specialized analysts that never sleep: one watches trends in stocks, another trades bonds, another protects against crashes, another trades crypto, and so on. Each analyst has its own job and its own slice of the portfolio. Right now, CHAD runs on practice money to prove it works before real dollars are committed.

Technically, CHAD is a multi-strategy algorithmic trading system built in Python 3.12 on an AWS EC2 m5.large instance in ca-central-1. It connects to Interactive Brokers for equities, futures, and options execution; Kraken WebSocket for real-time crypto pricing; and an Anthropic Claude AI intelligence layer for macro analysis, confidence biasing, sentiment analysis, and operator advisory. The system is architected around three strict planes — Intelligence, Trading, and Safety — with approximately 79 systemd services orchestrating market data ingestion, signal generation, execution planning, risk management, and observability. All runtime state is file-based with atomic writes (tmp+fsync+rename), accelerated by Redis pub/sub for sub-100ms signal propagation, and every component is fail-closed by design.

## 2. Architectural Principles

### 2.1 Three-Plane Model

**Intelligence Plane** — what CHAD thinks

Encompasses all AI-powered analysis: Claude API calls (3-tier routing), advisory synthesis (6 engines), strategy intelligence (confidence bias, universe filtering, regime classification), Google Trends signals, Reddit sentiment analysis, short interest data, macro interpretation, and Telegram advisory responses. Components in this plane can read any runtime state file but can write only to advisory-specific outputs (runtime/strategy_intelligence.json, runtime/trends_state.json, runtime/reddit_sentiment.json, runtime/short_interest.json, reports/advisory/). They cannot modify config files, operator intent, stop state, or any trading parameters. They produce recommendations — never actions.

**Trading Plane** — what CHAD does

Encompasses signal generation (14 strategies), execution planning (signal router, execution pipeline), broker communication (IBKR adapter with 5 lanes, Kraken), position management, and trade ledger recording. Components here read strategy configs and risk caps, generate TradeSignal objects, route them through LiveGate, and submit orders to the broker. The trading plane operates on deterministic rules — never on AI output directly. AI confidence adjustments are bounded [-0.15, +0.10] and clamped before application.

**Safety Plane** — what CHAD cannot do

Encompasses LiveGate (10+ independent gates), profit lock (5 escalating tiers), SCR (shadow confidence routing), stop state, operator intent, config snapshots with tamper detection, change canary, mutation state, reconciliation, and lifecycle truth. Every component in this plane is fail-closed: if data is missing, stale, or corrupt, the answer is always DENY. The safety plane has absolute veto over the trading plane. No trade can execute without passing all gates.

### 2.2 The Cardinal Rule

**Claude AI is intelligence plane only. It advises. It never executes.**

What Claude CAN do:
- Analyze market data and provide advisory text
- Adjust confidence bias within bounded range [-0.15, +0.10]
- Recommend universe filtering (avoid/prefer symbols)
- Classify market regime (risk_on/risk_off/neutral/crisis)
- Generate daily reports and morning briefs
- Answer operator questions via Telegram
- Produce research, macro, risk, and model health explanations

What Claude CANNOT do:
- Execute trades or submit orders
- Modify config files or strategy weights
- Change risk parameters or position limits
- Override LiveGate or any safety gate
- Approve or apply pending actions
- Modify operator intent or stop state
- Access API keys or credentials
- Write to any file outside advisory/intelligence outputs

### 2.3 Fail-Closed Philosophy

| Subsystem | Fail-Closed Behavior |
|-----------|---------------------|
| **LiveGate** | Missing or stale state file → DENY_ALL |
| **Stop state** | Missing or expired TTL (24h) → `stop=true` |
| **Operator intent** | Missing → DENY_ALL; default mode EXIT_ONLY |
| **SCR** | Missing → PAUSED (`paper_only=true`, sizing=0.00) |
| **Profit lock** | HARD_STOP tier → `stop_new_entries=true` |
| **Execution quality** | "dangerous" or "unknown" → DENY_ALL |
| **Reconciliation** | RED or missing → DENY_ALL |
| **Mutation state** | Pending actions exist → DENY_ALL |
| **Change canary** | Config drift detected → `canary_factor<1.0` → DENY_ALL |
| **Lifecycle truth** | `truth_ok=false` or `gap_flag=true` → DENY_ALL |
| **Strategy intelligence** | Claude unavailable → neutral bias (0.0), no filtering, "normal" regime |
| **Signal router** | Exception during routing → empty signal list |
| **IBKR adapter** | No IB session injected → ConnectionError (P0-3) |
| **Broker timeout** | >10 seconds → BrokerTimeoutError (P0-2) |
| **Strategy attribution** | Cannot resolve strategy → StrategyAttributionError (P0-4) |

### 2.4 Governance Pipeline

All configuration changes follow the Pending Actions pipeline. No direct file edits to governed config.

**Complete flow:**
1. Change is written as a JSON action file to `control/pending_actions/`
2. Operator reviews the action and moves it to `control/approved_actions/`
3. `chad-action-applier.service` scans approved queue
4. Pre-apply: config snapshot created (SHA256 hash of all config files)
5. Action applied atomically (tmp → fsync → rename)
6. Post-apply: action archived to `control/applied_actions/`
7. Mutation state publisher updates `runtime/action_state.json`
8. Change canary validates composite hash against pre-apply snapshot

**Supported action kinds:**
- `weight_rebalance_config` — atomically updates `config/strategy_weights.json` (validated sum 0.985–1.015)
- `rebalance_execute` — runs rebalance executor in subprocess

**Idempotency:** Each action has a unique ID tracked in `runtime/action_applier_state.json`. Applied action IDs stored in set; re-processing same ID is a no-op.

**LiveGate integration:** While `pending_count > 0` or `rejected_count > 0`, LiveGate returns DENY_ALL. No live trading during governance transitions.

### 4.2 Enforcement Mechanism

**File**: `chad/risk/dynamic_risk_allocator.py`

`enforce_chassis()` runs every allocation cycle:
1. Maps each strategy to its sleeve via lookup table
2. Sums actual allocation per sleeve
3. If any sleeve drifts > **5% tolerance** from target: proportional scaling applied
4. Intra-sleeve proportions preserved (strategies within a sleeve maintain relative ratios)
5. Unknown/unmapped strategies zeroed out
6. Final weights normalized to sum ~1.0

### 4.3 Why This Matters

The BETA sleeve is the safety net. It holds 25–30% of capital in long-term, low-churn positions that compound over time. Without chassis enforcement, aggressive alpha strategies could gradually consume this allocation during winning streaks, leaving the portfolio over-exposed when a correction hits. The 50/30/20 chassis ensures that no matter how well or poorly individual strategies perform, the portfolio always maintains structural diversification. It is the difference between a hedge fund and a gambling account.

### 4.4 Override

Set `CHAD_CHASSIS_ENFORCEMENT=0` to disable enforcement. Only during controlled testing where you need raw strategy weight behavior without chassis normalization. Never in production.

## 5. Execution Architecture

### 5.1 Signal Flow

```
Strategy handler (e.g., alpha.py)
    ↓ emits TradeSignal(strategy, symbol, side, size, confidence, asset_class, meta)
SignalRouter (chad/utils/signal_router.py)
    ↓ groups by (symbol, side, asset_class), merges sizes additively,
    ↓ confidence = size-weighted average, produces RoutedSignal
ExecutionPipeline (chad/execution/execution_pipeline.py)
    ↓ nets opposing BUY/SELL, validates prices > 0, builds PlannedOrder objects
LiveGate (chad/core/live_gate.py)
    �� evaluates all gates — ALLOW_LIVE, EXIT_ONLY, or DENY_ALL
IBKRAdapter (chad/execution/ibkr_adapter.py)
    ↓ normalizes to NormalizedIntent, routes by sec_type
ContractResolver
    ↓ resolves to ib_insync contract (Stock, Future, Option, Combo, Forex)
OrderFactory
    ↓ builds ib_insync Order (MKT/LMT, TIF, quantity)
Broker submission
    ↓ _call_with_timeout(ib.placeOrder, contract, order, timeout_s=10)
Fill confirmation
    ↓ SubmittedOrder with ib_order_id, status
NDJSON ledger
    → data/fills/FILLS_YYYYMMDD.ndjson (hash-chained, append-only, fcntl-locked)
```

### 5.2 Execution Lanes

**STK Lane** — Equities/ETFs: `sec_type=STK`, exchange=SMART, USD, whole units. No special meta required.

**FUT Lane** — Futures: `sec_type=FUT`, exchange per spec (CME/NYMEX/COMEX/CBOT). **Requires `contract_month` in meta** (P0-1 hardening — no hot-path reqContractDetails). Supported: MES, MNQ, MCL, MGC, ZN, ZB.

**OPT Lane** — Options: `sec_type=OPT`, exchange=SMART. **Requires `expiry`, `strike`, `right` in meta**. Multiplier defaults to 100.

**BAG Lane** — Combo Spreads (Phase 6b): `sec_type=BAG`, two ComboLeg objects (long BUY + short SELL). **Requires `expiry`, `long_strike`, `short_strike`, `long_right`, `short_right` in meta**. Atomic execution — both legs or neither.

**CASH Lane** — Forex: `sec_type=CASH`, exchange=IDEALPRO. 6-letter symbol parsed to base/quote. Fractional units (quantity_step=0.0001).

### 5.4 P0 Hardening (Completed April 2, 2026)

**P0-1: Remove reqContractDetails from hot paths** — Futures require `contract_month` in meta; options require `expiry/strike/right`. No broker socket calls during contract resolution. Raises `ContractResolutionError` if metadata missing.

**P0-2: 10-second execution liveness contract** — All broker calls wrapped in `_call_with_timeout(fn, timeout_s=10.0)` using daemon thread with hard wall-clock deadline. Failure classes: TIMEOUT, BLOCKED, REJECTED, FAILED, UNKNOWN. Applied to: qualifyContracts, whatIfOrder, placeOrder, connect.

**P0-3: Single IB() session discipline** — No implicit `IB()` creation in hot path. Adapter requires pre-managed session via `ib_factory` dependency injection. Default lazy factory raises `ConnectionError` if not injected.

**P0-4: Fix paper_exec attribution** — Never silently flatten strategy to "paper_exec". Raises `StrategyAttributionError` if strategy cannot be resolved. Attribution precedence: explicit kwarg → source_strategies → planner artifact → tags → error. Placeholder names ("", "unknown", "manual", "paper_exec") never accepted as primary.

## 9. Governance and Safety

### 9.1 Pending Actions Pipeline

**Directory structure**: `control/` with subdirectories: `pending_actions/`, `approved_actions/`, `applied_actions/` (8 completed), `expired_actions/` (11 expired), `rejected_actions/`, `unsupported_actions/`.

**Flow**: Create JSON → operator reviews → move to approved → ActionApplier dispatches → pre-apply config snapshot → validate (weights sum 0.985–1.015) → atomic write → archive to applied → mutation state published → change canary validated.

**Action kinds**: `weight_rebalance_config` (atomic strategy_weights.json update), `rebalance_execute` (subprocess rebalance).

**Idempotency**: `runtime/action_applier_state.json` tracks applied action IDs in a set. Re-processing same ID = no-op. State includes: last_apply_ts_utc, last_action_id, last_receipt_path, applied_actions list, applied_action_meta dict.

### 9.2 Config Snapshots

**File**: `chad/ops/config_snapshot.py`

- Snapshots all `*.json` files in `config/` (~15 files)
- Per-file SHA-256 hash + size_bytes
- **Composite hash**: SHA-256 of concatenated sorted "filename:hex_hash" pairs
- Timestamped output: `data/config_snapshots/snapshot_YYYYMMDDTHHMMSSZ.json`
- Pre-apply snapshot before every action (historical record)
- Periodic comparison by mutation_state_publisher
- **Tamper detection**: Unexpected composite hash change → change_canary_state.json sets canary_factor < 1.0 → LiveGate DENY_ALL

### 9.3 Profit Lock

**File**: `chad/risk/profit_lock.py` — Five escalating tiers:

| Tier | PnL Threshold | Sizing Factor | Stops New Entries |
|------|--------------|---------------|-------------------|
| NORMAL | < 1.5% | 1.00 | No |
| WARN | >= 1.5% | 1.00 | No (alert only) |
| LOCK1 | >= 3.0% | 0.50 | No |
| LOCK2 | >= 5.0% | 0.25 | No |

## 11. Known Limitations and Deferred Items

### 11.1 Active Limitations

1. **Paper-only execution** — Real broker behavior (slippage, partial fills, market impact) not validated
2. **Daily bar backtesting** — Intraday dynamics, gap risk, micro-structure not modeled
3. **No slippage model** — Paper fills at close price; real execution has spread, latency, impact
4. **Bootstrap configs** — risk.json, data_sources.json, feature_flags.json contain placeholders
5. **Options chain data** — alpha_options needs real-time IBKR options subscription
6. **M6E subscription** — omega_macro's Micro Euro FX may need separate data subscription
7. **SCR sample** — 1,515 effective trades; statistically encouraging but not conclusive
8. **Reddit API** — Public JSON endpoint may rate-limit; PRAW with credentials would improve coverage

7. **Correlation breakdown** — Delta_pairs assumes stable 0.94–0.99 correlations
8. **SCR calibration** — Thresholds and sizing factors may need live adjustment

## 13. Glossary

| Term | Definition |
|------|-----------|
| **ATR** | Average True Range — how much a price typically moves per bar. Higher = more volatile. |
| **BAG order** | A combo order bundling two options legs into one atomic trade. Both execute or neither does. |
| **Bollinger Bands** | Bands drawn 2 standard deviations above/below a moving average. Price at the edge = extended. |
| **Canary** | Change-detection mechanism. If config files change unexpectedly, the canary triggers and blocks trading. |
| **Chassis** | The 50/30/20 allocation framework protecting portfolio structure. Like a car frame — keeps everything in place. |
| **Config snapshot** | Timestamped SHA-256 hashes of all config files. Detects unauthorized changes. |
| **Confidence** | A 0–1 score representing how strongly a strategy believes in a trade. Affects position size. |
| **Conviction score** | Multi-factor 0–1 score used by delta strategy combining vol expansion, breakout, momentum, event risk. |
| **DecisionTrace** | Append-only audit log recording every orchestrator decision per cycle. For debugging and compliance. |
| **EMA** | Exponential Moving Average — smoothed price average giving more weight to recent data. |
| **Fail-closed** | Design where missing or broken data → safest possible action (deny trading, reduce size). Opposite of fail-open. |
| **Half-life** | In pairs trading, expected days for a diverged ratio to return halfway to its mean. Shorter = faster reversion. |
| **Hash chain** | Sequence where each record contains the hash of the previous record. Proves no records tampered after the fact. |
| **Idempotency** | Guarantee that submitting the same order twice produces only one execution. Tracked via SQLite. |
| **LiveGate** | 10-gate safety system that must approve every live trade. All gates must pass. One failure = all blocked. |
| **NDJSON** | Newline-Delimited JSON — each line is a complete JSON object. Used for trade ledgers and audit logs. |
| **Operator intent** | File-based control telling CHAD whether live trading is allowed. Modes: DENY_ALL, EXIT_ONLY, ALLOW_LIVE. |
| **Pending Action** | Proposed config change in a queue until a human approves. No changes bypass this process. |
| **Point value** | Dollar value of one point of price movement in a futures contract. MES=$5, MCL=$100. |
| **Profit Lock** | Five-tier system that reduces position sizes (and eventually stops trading) as daily profits grow. |
| **Reconciliation** | Comparing what CHAD thinks it holds vs. what the broker reports. Must be GREEN for live trading. |
| **Regime** | Current market environment: risk_on, risk_off, neutral, or crisis. Different strategies perform differently. |
| **RSI** | Relative Strength Index — 0–100 momentum indicator. Above 70 = overbought. Below 30 = oversold. |
| **SCR** | Shadow Confidence Router — monitors CHAD's trading performance and adjusts sizing based on win rate. |
| **Sleeve** | Bucket within the 50/30/20 chassis. ALPHA (growth), BETA (preservation), ADAPTIVE (tactical). |
| **Squeeze risk** | When heavily shorted stock rises, shorts must buy to cover, pushing price higher in a feedback loop. |
| **Stop state** | Emergency kill switch. When engaged, all trading halts immediately. |
| **TradeSignal** | Standard output from every strategy: symbol, side, size, confidence, asset class, metadata. |
| **TTL** | Time To Live — how long data is considered fresh. After expiry, system treats it as stale and may fail-closed. |
| **VIX** | CBOE Volatility Index — the "fear gauge." Below 15 = calm. Above 25 = worried. Above 30 = panic. |
| **Z-score** | Statistical measure of how far a value is from its average in standard deviations. Z=2 = very unusual. |
