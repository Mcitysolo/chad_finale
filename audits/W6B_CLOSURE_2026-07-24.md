can be computed or compared". The ML_SHADOW lines are fully structured, so a rate *is*
computable — measured **0.19% (1/526)** against the manifest's training-time **71.2%**, a

fails closed.

## 4. Where the design departed from the plan, and why

**EXS7's filter is not about freshness.** The plan proposed "live publisher + known
cadence". Reading `check_schema_breaks` showed it breaks on `missing | unreadable |
version-absent/unrecognised | required_keys_missing` — **staleness is not among them**.
`savage_alloc_state.json` has not been written in 136 days and would pass every cycle. The
real disqualifier is *"can this file be legitimately ABSENT during normal operation?"*,
which yields three exclusion classes instead of an age cutoff.

**D1's envelope means `position_guard.json` is never touched.** Making `schema_version`
validation opt-in lets a contract be validated on its envelope alone, so no key enters the
identity namespace and the reader hazard the plan flagged never arises. The plan's fallback
was "document as a justified exception, do not pin"; this gets real enforcement at the same
zero risk.

**The envelope rule is conditional, and the live data hid why.** All 39 current entries
carry `opened_at`, so an envelope inferred from observation would look correct *and pass
the pre-flight*. But `position_guard.py:459` builds closed entries without it. Deriving the
rule from the writer's own declared invariant — *"closed entries can be sparse"* — is what
avoids turning a recovery tool into a sentinel failure.

**Baseline sufficiency needed a second gate.** My first version used sample count alone,
and the live data immediately showed why that fails: 526 rows gathered in **5.26 hours** of
one boot passes any n floor and is still one session, 94% one strategy. Sufficiency now
requires both n ≥ 200 and span ≥ 72h — which is why W6B-6 is correctly **blocked**.

## 5. Two defects the tests caught during the build

Both in W6B-13, both the exact failure class that tool exists to prevent:

1. **`complete` after cancelling nothing.** The outcome was `not failed and len(cancelled)
   == len(cancellable)` — with nothing cancellable that is `0 == 0`, so an execute run that
   cancelled zero orders reported **"complete"**. Zero cancels is now always
   `nothing_cancelled`, with a loud note that it is *not proof the book is clear*.
2. **Unparseable orders vanishing.** A broker object of unexpected shape sailed through the
   `getattr` defaults into an all-zero `OrderRow` — still left alone, but absent from
   `orders_unparseable`, so an operator would never learn the broker returned something the
   tool did not understand.
